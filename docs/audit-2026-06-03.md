# Audit — 2026-06-03 (code / security / privacy / workflow)

Fourth audit pass, focused on the newly-shipped **at-rest encryption** plus a broad
sweep. Run as four parallel read-only lenses (code/bug, security, privacy, workflow).
Baseline going in: clippy warning-clean, 184 tests green, crypto core sound, every
config-dir store routes through the sealed writer, all network egress honors the
SOCKS5 proxy. This doc records what was found and what was done.

## Fixed

### HIGH
- **TOCTOU on concurrent enable/disable/change-password → vault corruption.** Each
  migration handler checked vault state once before the KDF and never re-checked under
  the migration lock, so two rapid requests could double-seal files and lose data.
  Fix: a process-global claim (`at_rest::try_claim_migration` / `MigrationClaim`,
  `MIGRATION_IN_PROGRESS` AtomicBool) taken at handler entry; a second concurrent
  request gets `400 "another encryption change is already in progress"`. Test
  `migration_claim_is_exclusive_and_frees_on_drop`. (`at_rest.rs`, `api/security.rs`)
- **Onboarding Escape silently skipped encryption setup.** Escape on the "Protect your
  data" step marked onboarding complete (and it never reappears), discarding the
  born-encrypted opportunity. Fix: Escape no longer skip-completes on step 3.
  (`OnboardingWizard.svelte`)
- **Delete-wallet confirm had no busy state** → double-`DELETE` + frozen-looking modal.
  Fix: the standard `{busy ? 'Deleting…'}` + `disabled` pattern, awaiting the parent.
  (`DeleteWalletModal.svelte`)

### MEDIUM
- **Send drafts + multisig setup persisted sensitive data to `localStorage`** (the
  webview profile, *outside* at-rest encryption): recipient address + multisig signer
  xpubs. Fix (user decision: drop sensitive fields): drafts now persist amount / fee /
  coin-control / threshold but never the recipient address or signer xpubs; older
  drafts have them stripped on load. (`send/SendFlow.svelte`, `MultisigCreatePanel.svelte`)
- **`/api/status` was reachable while the vault was locked** and would probe the
  default public Electrum server pre-unlock, un-proxied (latent — the frontend doesn't
  call it while locked). Fix: removed from the lock-gate allow-list. (`lib.rs`)
- **Unlock gate gave no guidance after repeated failures.** Fix: after 3 failures it
  shows the no-recovery reality (re-add wallets from seed; money is safe). (`UnlockGate.svelte`)

### LOW
- **AAD bound to the basename, not the relative path** (a latent file-swap footgun if a
  future store reuses a basename in a subdir). Fix (user decision: clean break): AAD is
  now the config-root-relative path (`at_rest::file_aad` + `set_config_root`, migration
  uses `aad_for_relpath`). **Format change — an existing encrypted vault must be
  disabled before updating, then re-enabled.** Test `relpath_aad_binds_location_not_just_name`.
- **`read_sealed` didn't take the migration lock** → a read in the post-swap /
  pre-vault-flip window could misread. Fix: `read_sealed` now holds `MIGRATION` shared.
- **SP scanner relaxed TLS hostname verification when `validate_tls` was off**,
  inconsistent with the Electrum native-tls path. Fix: only relax under the explicit
  `danger_accept_invalid_certs`. (`sp_scanner.rs`)
- **SP subscriber logged the receive address (INFO) + a txid (WARN)** — the only
  wallet-identifying data in default logs. Fix: demoted both to DEBUG. (`sp_subscriber.rs`)
- **ReceiveModal cancel-payjoin wasn't guarded; bare `clipboard.writeText` lacked
  `.catch`.** Fix: gate cancel on `pjrBusy`; clipboard writes catch + toast.
- **`/unlock` ran Argon2 without serialization** → parallel requests could run N hashes
  at once, defeating the cost as a rate limit and pinning N×64 MiB. Fix: a global async
  gate serializes unlock attempts (one 64 MiB hash at a time; the single user never
  contends). A *hard lockout* is still intentionally not added (would risk locking out
  the legitimate user; the serialized Argon2 cost is the throttle). (`api/security.rs`)

## Documented (can't be fully fixed in code)
- **Webview-profile data outside the boundary.** `localStorage` (now only non-sensitive
  prefs after the fix) and the WebKit/Tauri profile aren't covered by at-rest
  encryption. Noted in the help article and `at-rest-encryption.md`.

## Considered, deliberately not done
- **A hard unlock lockout** (after N failures) — would risk locking out the legitimate
  user with no recovery; the serialized Argon2 cost (added above) is the throttle.
- **SP `scan_secret` as `Zeroizing` in memory** — the scan secret exists in several
  in-memory forms (a `[u8;32]` field *and* a `scan_secret_hex` String held by the
  persisted store and the running scanner tasks across ~5 files). Wrapping only the
  field is theater while the String copies remain; wrapping all of it is a
  disproportionate ripple for a secret that's persisted (now encrypted) anyway under a
  trusted-local threat model.
- **Two-phase unlock label** ("Unlocking…" → "Starting wallets…") — the backend does the
  KDF *and* the full startup in one request, so a real two-phase cue needs splitting the
  request (return after KDF, start services async). That's an architecture change, not a
  label tweak; not worth it for a sub-second cue on small wallets.
- **enable-error `e.code` branching** — `enable` has no `wrong_secret` path (its only 4xx
  is "already enabled" = `bad_request`), so branching on `e.code` would add a dead arm;
  showing `e.message` is correct.

## Checked and solid (no finding)
- At-rest AEAD/KDF: Argon2id → HKDF-SHA256 domain-separated subkeys → XChaCha20-Poly1305,
  random nonce per write, tamper/truncation/wrong-key/wrong-AAD all rejected, salt
  random per install, params in the sentinel, keys `Zeroizing` (zeroize on drop/disable).
- Migration crash-safety (staging + two-rename swap + boot recovery), boot-locked gating
  (no disk read / no subscriber while locked), disable + change-password re-verify the
  password against the sentinel.
- Money paths: SP-spend key-path Schnorr signing + sanity checks, regular send / RBF /
  CPFP fee math (`saturating_*`), snapshot Arc-swap concurrency, no bare `unwrap()` in
  production. Send v2 verify-before-broadcast still independently re-checks the signed
  tx's recipient against intent.
- All network egress proxied; per-wallet backend resolution correct; service worker
  never caches `/api/*`; secrets (`Zeroizing`) not logged; prior audits' fixes hold
  (whole-wallet-consolidation, BIP-353 Quad9, change-script-mismatch + SP input warnings).
