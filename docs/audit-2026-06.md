# Corvin audit — 2026-06

Second in-depth pass (the first was `docs/audit-2026-05-31.md`), reading actual code
across the money-out core, then the surface the first pass didn't deeply cover. The
privacy lens has its own writeup in `docs/privacy-audit.md`; this file is the
bug / security / performance record. **All findings below are fixed**, and the
fund-safety ones are locked by tests (see "Test coverage added").

## Headline

One **high-severity** bug: the default send path force-spent the entire wallet on
every transaction (privacy + fee disaster). Everything else was low/medium hardening.
Secret handling, atomic 0600 writes, the loopback/CORS boundary, and SOCKS5 proxy
coverage on every egress path were all re-verified clean.

## Findings ledger (all fixed)

| # | Sev | Area | Finding | Fix |
|---|-----|------|---------|-----|
| 1 | 🔴 HIGH | `send.rs`, `sp_send.rs` | Auto-select (no coin control = the default) added **every** eligible UTXO via `add_utxo` + `manually_selected_only`, force-spending the whole wallet each send — links all coins on-chain, bloats fees. | Compute the *unspendable* set (frozen ∪ immature ∪ timelocked) → `builder.unspendable(...)` and let BDK coin-select. Send-max adds `drain_wallet()` (still respects `unspendable`). |
| 2 | 🟡 MED | `sp_spend.rs` | Fee rate unvalidated — a non-finite/huge value could overflow to `u64::MAX`. | Route through `validated_fee_rate_sats` (rejects NaN/Inf, caps 10k sat/vB). |
| 3 | 🟢 LOW | `sp_spend.rs` | Drain output only guaranteed ≥ 1 sat — could build a sub-dust output. | Require drain ≥ `DUST_SATS` (330). |
| 4 | 🟢 LOW | `sp_spend.rs` | When dust change is folded into the fee, the reported `fee_sats` understated the actual fee. | Report the real fee (input − recipients). |
| 5 | 🟢 LOW | `state.rs` | A corrupt annotation file (labels/categories) silently started empty, and the next write overwrote it. | `tracing::warn!` on parse failure. |
| 6 | perf | `send.rs` | `compute_send_warnings` scanned the full tx history twice per preview. | One pass feeds both repeat-recipient + look-alike. |
| 7 | 🟡 MED | `crud.rs`, `sp_subscriber.rs` | Deleting an SP wallet did **not** abort its scanner task — the scan secret lingered in RAM, the third-party SP server stayed subscribed, and the scanner could resurrect the just-deleted `sp_outputs.json`. | Abort `state.sp_scanners[id]` in `delete_wallet`, before clearing keys. |
| 8 | 🟢 LOW | `silent_payments.rs` | The SP key store chmod'd 0600 *after* the rename, leaving a brief world-readable window on the scan secret. | chmod the temp file *before* rename (the `write_private` pattern). |
| 9 | 🟢 LOW | `messages.rs` | The BIP-322 derived WIF (a private key) wasn't zeroized, unlike the seed/passphrase. | Wrap in `Zeroizing`. |

## Surface newly audited (verified clean — no findings)

The 2026-05-31 pass focused on the request/validation layer; this pass added:

- **`descriptor.rs`** — path/fingerprint validation prevents descriptor injection;
  everything round-trips through miniscript; the timelock policy logic
  (`timelock_spendable`, vault-vs-savings detection) is correct.
- **`backup_crypto.ts`** — Argon2id (m=19 MiB/t=3, OWASP-compliant) + AES-256-GCM
  with a fresh salt→key→IV per backup (no nonce reuse), decrypt-time param bounds,
  key bytes zeroed after import.
- **Payjoin (send + receive + subscriber)** — leans on the crate's typestate, so the
  BIP-77 receiver checks can't be skipped; the receiver's fee exposure is bounded by
  `max_fee_rate = BROADCAST_MIN`; the sender's fallback tx is always broadcastable.
- **`subscriber.rs`** — workers are isolated (own their maps), no shared mutable
  state without locks, syncs serialize on the per-wallet mutex.
- **`sp_spend.rs` signing** — key-path Schnorr with parity handled by `sign_schnorr`,
  per-input sanity check that `d_n` controls the recorded output key.

### Third pass (the "lighter-covered" tail, now line-by-line)

- **HW stack** (`ledger.rs`, `bitbox.rs`, `trezor.rs`, `hwi/common.rs`) — sound trust
  model across all three brands: the connected device's master fingerprint is verified
  against the wallet's expected one *before* signing; addresses/txs are confirmed
  on-device; sigs are merged by input index but only finalize if miniscript verifies
  them, so a wrong/invalid sig can't produce a broadcastable tx. Finalization
  (`hwi/common.rs`) is unit-tested. (Trezor multisig signing is a documented
  not-yet-supported limitation, not a bug.)
- **`multisig.rs` combine** — checks the txid matches before merging; `ready` is true
  only if `finalize_mut` verifies the sigs against the descriptor; Coldcard export
  filename is injection-safe.
- **`bip329.rs`** — bad lines skipped, unknown types ignored, labels render as escaped
  Svelte text (no stored-XSS), bogus refs are harmless.
- **`import.ts`** — hints-only and backend-revalidated (descriptors round-trip through
  miniscript server-side); parsers correct, and already covered by `import.test.ts`
  (23 tests across all four parsers).
- **`tax.rs`** — lots tracked in integer sats (no float drift), lot state maintained
  across years. **Accuracy nuance (LOW, advisory feature) — fixed:** cost-basis
  overrides now stay consistent with the lot pool — an override on a *received* tx sets
  that lot's basis (so it flows into later sales), and a *sent* tx always consumes
  inventory (even with an override on the reported basis), so coins aren't
  double-counted. Lot math (`consume_lots` FIFO/LIFO/HIFO + partial/over-consume) is
  now unit-tested.

No remaining backend area is unaudited. Frontend Svelte components (vs. pure `lib/`)
are still only type-checked, by design.

## Test coverage added

A hermetic regtest harness in `crates/server/src/api_tests.rs` (`funded_regtest_wallet`)
funds a real BDK wallet in memory and drives the live handlers — no bitcoind/electrs,
since the build path never touches the network. New tests:

- **Coin selection (#1):** regular auto-send doesn't consolidate; send-max drains all;
  coin control spends exactly the picked inputs; SP-send auto doesn't consolidate.
- **SP-spend signing (#2–4):** `build_sp_spend_tx` (extracted from the handler for
  testability) — the hand-rolled key-path Schnorr signatures are **verified to
  validate** against each input's output key; sub-dust drain is rejected.
- **Privacy warnings:** all six codes (mixed labels/categories, repeat recipient,
  round amount, look-alike via pure unit tests, change-script-mismatch).
- **Lifecycle (#7):** deleting a wallet aborts its SP scanner.
- **Backup (#10):** a freshly-exported backup round-trips through restore.

Server lib tests went 19 → 37 over the audit-fix-test cycle (incl. `consume_lots`
FIFO/LIFO/HIFO unit tests). `cargo clippy --workspace --all-targets` clean;
`svelte-check` 0/0; 95 frontend Vitest tests pass.
