# Corvin audit — 2026-05-31

In-depth pass across security, bugs, UX/UI, memory, and resource usage, reading
actual code + running tooling. Scope: `crates/` (~16k LOC Rust) + `frontend/src`
(~23k LOC). Headline: **the codebase is in good shape** — clippy/tests/audit clean,
careful input validation, correct secret handling, no panics in request paths, no
memory leaks found. Findings below are mostly low-severity hardening + polish.

## What's done well (verified, not assumed)

- **Secret handling.** Mnemonics/passphrases arrive as plain `String` in request
  structs but are `std::mem::take`-moved into `Zeroizing` as the first lines of each
  handler (messages, sp_send, payjoin, backup_test, etc.), wiping the heap buffer on
  every return path. Seed/spend-secret derivation in `core/silent_payments.rs` and
  `seed.rs` uses `Zeroizing` throughout. **No secrets logged** (grepped all log
  macros). Seeds never persisted.
- **No-auth boundary.** Loopback enforced at *startup* (`lib.rs` re-checks the
  persisted/hand-edited bind and falls back to 127.0.0.1 unless `CORVIN_ALLOW_LAN`),
  not just on settings-save. CORS allowlist restricts to localhost origins.
- **Input validation.** Proxy txid strictly checked (`len==64 && all hex`) before URL
  interpolation — no SSRF. Path params are `Path<Uuid>` for wallet/file paths (UUID
  can't traverse); `Path<String>` cases are HashMap keys, never filenames.
- **No request-path panics.** Audited every non-test `unwrap`/`expect`/`unreachable`:
  all are guarded (prior `is_some()`/length checks), poison-lock unwraps, or thread-
  start. Float→fee cast overflow (`Infinity`) is caught by `FeeRate::from_sat_per_vb`
  returning `Option` + `?`. **Now enforced by lint, not just audit:** the
  `clippy::unwrap_used` ratchet (workspace, warn; tests exempt via `clippy.toml`)
  flags any new bare `unwrap()` in production — the prior sites were cleaned to
  poison-tolerant lock reads, `if-let`/`flatten` restructures, or `.expect("why")`.
- **Money math authority.** Client BTC→sats uses JS float (`Math.round(n*1e8)`), but
  the server takes `u64` sats and BDK's `.finish()` enforces dust/funds/fee at build
  time — client value is never authoritative. Float concern is cosmetic only.
- **Async correctness.** All `blocking_lock()` calls are inside `spawn_blocking`
  closures (no async-context blocking). SSE handles `Lagged` (skip, stream survives).
  SP scanner aborts the old handle before respawn (no task leak), exponential backoff
  5s→300s, network-mismatch guard.
- **Frontend memory.** All `setInterval` cleaned (onDestroy / `$effect` return). Only
  5 manual `addEventListener` in script; all either return a remover or are
  intentional app-lifetime (theme.ts). SendModal/QrSignFlow/etc. clean timers +
  EventSource on destroy. The 62-vs-11 add/remove gap is `<svelte:window>` directives
  (auto-cleaned), not leaks.

### Low / hardening

- **F-1 — DNS-rebinding / no Host-header check. [ACCEPTED, not fixing.]** CORS
  validates `Origin`. A DNS-rebind (attacker hostname → 127.0.0.1, Origin appears
  allowed) is the theoretical gap, BUT every state-changing endpoint takes a
  `Json<>` body → requires `Content-Type: application/json` → forces a CORS preflight
  the rebound origin fails. So **writes are already blocked**; a rebind could at most
  do **GET reads** (balances/xpubs — privacy leak, no fund/config impact), needs a
  predictable port + victim visiting an attacker page, and is moot on reverse-proxied
  Start9. Decided 2026-05-31: real-but-tiny, read-only, low-probability — left as
  accepted risk. A Host-header allowlist middleware remains an option if it ever
  matters.
- **F-2 — `fee_cache` never pruned.** `ManagedWallet.fee_cache: HashMap<txid,fee>`
  only grows via `.extend()`; stale/replaced txids accumulate for the process
  lifetime. In-memory only (not on disk), so bounded by uptime — a long-running Start9
  wallet with churn leaks slowly. **Fix:** prune to current canonical txids on sync,
  or cap size.
- **F-3 — fee-rate not finite/upper-bounded.** `fee_rate_sat_vb.max(1.0).ceil() as
  u64` turns `Infinity` into `u64::MAX` (then caught by `from_sat_per_vb`). NaN/neg
  are handled (→1). Not exploitable (no auth = only hurts self), but **add an explicit
  `is_finite()` + sane upper bound** (e.g. ≤ a few thousand sat/vB) for a clean error.

### Informational / polish

- **F-4 — fee_cache lock churn.** `background_sync` locks `fee_cache` 5× per sync
  (clone/extend/clone). Correct but wasteful; collapse to one acquisition.
- **F-5 — JS sat precision.** `parseFloat(btc)*1e8` can mis-round exotic 8-dp values
  in the input display. Server is authoritative so no fund risk; consider parsing
  BTC→sats as integers (string split on '.') to avoid any displayed-vs-sent mismatch.
- **F-6 — guarded-unwrap comments.** `wallet.rs:221` (block_height.unwrap, guarded by
  filter) and `descriptor.rs:101` (try_into, guarded by len check) are safe but would
  read better as `expect("guaranteed by …")` or restructured.
- **F-7 — `cargo audit` not in CI. [BLOCKED — on hold.]** Root blocker: the project
  is **not a git repo and has no remote**, so there's nothing for CI to run on.
  Decided 2026-05-31 to hold until git is initialized + a host/remote is chosen (a
  positioning decision — public repo ties into the open-source/repro-builds trust
  story). When that lands, F-7 folds into **reproducible-builds Phase 1** (same CI
  bring-up: git + remote + pinned toolchain + workflow). The checks a workflow would
  run all exist + pass today: `cargo fmt --check` / `clippy -D warnings` / `test` /
  `audit` (+ `deny`), and frontend `npm run check` (svelte-check) / `npm test`
  (vitest) / `build`. Plus the GTK/webkit system deps for the Tauri link step.
- **F-8 — no automated frontend tests. [DONE — see update banner.]** Vitest covers the
  pure `lib/` logic (amount/search/import/bip39/utils). The SendModal caveat is now
  largely addressed: its trickiest address/URI logic (shape detection, BIP21/scan
  extraction, BIP-353 HRN detection, address chunking) was extracted to a pure,
  unit-tested `src/lib/send.ts` (21 tests). Remaining SendModal logic is still
  stateful/component-coupled and manually tested; a deeper component-tree split was
  scoped but deferred as higher-risk.
- **F-9 — 3 clippy style lints.** `backends.rs:13` empty line after doc comment;
  `subscriber.rs:215` run_session 8/7 args; `state.rs:769` `repeat().take()`. Trivial.
- **F-10 — never run through rustfmt. [FIXED.]** A later cleanup pass found the Rust
  code had 467 fmt-diff hunks across ~48 files (hand-formatted, consistent, just not

  gate to keep it so.
